# Introduction

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

The `dispatcharr-addons` app currently bundles five unrelated containers — `ecm`, `game-thumbs`, `identifiarr`, `teamarr`, and `kptv-fast` — into a single multi-container HelmRelease sharing one PVC (`dispatcharr-addons`, 10Gi). This plan splits each container into its own dedicated Flux Kustomization and HelmRelease. Data is preserved by cloning the existing `dispatcharr-addons` restic/volsync backup into per-app repositories, restoring into new PVCs, and cleaning up leftover subdirectories afterwards.

## 1. Requirements & Constraints

- **REQ-001**: Each app (`ecm`, `game-thumbs`, `identifiarr`, `teamarr`, `kptv-fast`) must become an independent Flux Kustomization and HelmRelease in `kubernetes/apps/equestria/pvr/<appname>/`.
- **REQ-002**: All persistent data currently stored in `dispatcharr-addons` PVC subpaths must be preserved in the new per-app PVCs before the old app is removed.
- **REQ-003**: Volsync backups must be configured for each new app that requires a PVC: `ecm`, `game-thumbs`, `identifiarr`, `teamarr`.
- **REQ-004**: `kptv-fast` uses only an `emptyDir` `/tmp` mount — it does NOT require a PVC or volsync.
- **REQ-005**: The `identifiarr` app must retain its `chown-dbdir` and `identifiarr-db` init containers.
- **REQ-006**: All `ApplicationDefinition` resources must be migrated to their respective new app directories with `${APP}` matching the new app name (no longer prefixed with `dispatcharr-addons-`).
- **REQ-007**: `flux-local test` must pass before pushing any changes.
- **CON-001**: This is a production cluster — both the old and new apps must NOT run simultaneously (shared PVC would cause data corruption).
- **CON-002**: The restic repository path for each new app is `/repository/<appname>` (matches the volsync `ExternalSecret` template `RESTIC_REPOSITORY: "/repository/${APP}"`).
- **CON-003**: Do not edit files generated by `Update.cs`.
- **GUD-001**: Follow the existing ks.yaml pattern: `metadata.name/namespace` anchors, `commonMetadata.labels`, `dependsOn` on `truenas-volumes`, `volsync`, and `tailscale-operator` where needed.
- **PAT-001**: Use the `../../../../components/volsync` component in `ks.yaml` for all apps requiring persistent storage.
- **PAT-002**: Use `../../../../components/failover/fast-node-eviction` component in all `ks.yaml` files.

## 3. Alternatives

- **ALT-001**: Live-copy PVC data using a Kubernetes Job instead of restic clone — mount the old `dispatcharr-addons` PVC and `rsync` each subpath to a new PVC. Rejected because it requires the old HelmRelease to be suspended and new PVCs to be pre-created manually, which is more error-prone than the restic copy approach.
- **ALT-002**: Keep a single HelmRelease with multiple controllers (one per app) — reduces GitOps overhead but doesn't achieve true isolation (shared pod lifecycle, still one PVC). Rejected as it doesn't meet the goal of distinct kustomizations and helm releases.
- **ALT-003**: Restore each new app from the same `dispatcharr-addons` restic repo with APP-specific subPath filtering — restic does not natively support per-path restore to separate repos. Rejected in favor of cloning the full repo and cleaning up post-restore.

## 4. Dependencies

- **DEP-001**: Volsync operator must be running (`volsync` kustomization in `volsync-system` namespace).
- **DEP-002**: 1Password Connect must be available (`external-secrets` kustomization) to provision per-app volsync ExternalSecrets. The existing `Volsync Password` 1Password item must be reachable.
- **DEP-003**: Restic access to the Backblaze (or configured) backend with credentials from the existing `dispatcharr-addons-volsync-secret` Secret.
- **DEP-004**: `truenas-volumes` kustomization must be ready (provides `truenas-media` PVC reference).
- **DEP-005**: `tailscale-operator` kustomization must be ready (referenced in `dependsOn`).

## 7. Risks & Assumptions

- **RISK-001**: Restic copy may fail if the new repo path does not yet have an initialized repository — restic copy initializes the destination repo automatically, but the backend credentials must allow write access.
- **RISK-002**: Downtime during migration — `dispatcharr-addons` must be suspended before new apps deploy (CON-001). Estimate: 5–15 minutes while ReplicationDestinations restore PVCs.
- **RISK-003**: The cloned backup contains the full `dispatcharr-addons` PVC tree. Each new app will initially have extra subdirectories in its PVC. These are inert (apps use specific mount paths) but consume space until cleaned up in Phase 5.
- **RISK-004**: `identifiarr-db` init container requires internet access to `epg.jesmann.com` — verify the cluster's egress policy permits outbound HTTPS from namespace `equestria`.
- **RISK-005**: `game-thumbs` cache data is not critical — if the restore from the cloned repo fails or is stale, the app will rebuild its cache. This is an acceptable degraded state.
- **ASSUMPTION-001**: The restic backend for `dispatcharr-addons` is the same Backblaze bucket used by all other volsync apps in the cluster; credentials in `dispatcharr-addons-volsync-secret` are sufficient to initialize new repos at sibling paths.
- **ASSUMPTION-002**: Each new app name (`ecm`, `game-thumbs`, `identifiarr`, `teamarr`, `kptv-fast`) is unique in the `equestria` namespace and does not collide with existing PVCs or HelmReleases.
- **ASSUMPTION-003**: The `media` PVC (`truenas-media`) is a shared read-only/read-write-many claim and does not require per-app isolation.
- **ASSUMPTION-004**: PVC sizes: ecm=2Gi, game-thumbs=5Gi, identifiarr=5Gi, teamarr=2Gi. Adjust `VOLSYNC_CAPACITY` if actual data size differs.
